# Atom Contribution Contract

This document defines the shared contribution bar for the sibling `sciona-atoms*`
repositories. It is the baseline contract for atom PR qualification.

Repo-specific `CONTRIBUTION.md` files may add stricter domain rules, but they
should not weaken the requirements here.

## Scope

This contract applies to:

- wrapper atoms under `src/sciona/atoms/**`
- witness modules and witness exports
- probe catalogs under `src/sciona/probes/**`
- shared metadata sidecars such as `references.json`, `uncertainty.json`,
  `heuristic_metadata.json`, and `data/hyperparams/manifest.json`
- family heuristic assets under `data/heuristics/**`

Two implementation shapes are currently valid:

- contract-bearing wrappers: atom modules that use `@register_atom(...)`,
  `@icontract.require`, and `@icontract.ensure`
- pure runtime/expansion modules: typed public helpers that may not use the
  witness system, but still need honest interfaces, tests, and documentation

## Merge Gate

Before opening a PR to merge a new atom or atom-family change, the repo must
pass the shared verification scripts in this repo:

```bash
python scripts/verify_contribution_rules.py --repo-root .
python scripts/validate_dejargon.py --root .
pytest -q tests/test_verify_contribution_rules.py tests/test_validate_dejargon.py
```

Domain-specific repos should add their own tests and stricter validation on top
of this shared gate.

## Required Standards

### 1. Interfaces

Every public atom-facing function must:

- be fully type-annotated
- have an honest docstring describing the operation, inputs, and outputs
- avoid public `*args` and `**kwargs`
- avoid `Any` in the public interface unless the repo-specific contract clearly
  justifies it and review calls it out explicitly

Contract-bearing wrapper modules must additionally:

- use `@register_atom(...)` when the wrapper participates in the witness system
- include meaningful `@icontract.require` preconditions
- include meaningful `@icontract.ensure` postconditions
- keep `@register_atom(...)` outermost when present
- avoid local shadow witness stubs that override real witness imports

Witness modules must:

- expose explicit witness signatures
- use conservative abstract ghost types
- stay pure and side-effect free
- mirror the public atom surface honestly

### 2. Hyperparameters

If a repo exposes tunable parameters, it must keep a reviewed
`data/hyperparams/manifest.json`.

Approved tunables must record:

- atom identity and source path
- audit status
- explicit tunable parameter records
- defaults and bounded domains
- semantic role
- range/source provenance
- reviewer rationale

Do not expose hyperparameters by mechanically scraping signatures. The manifest
is a reviewed claim about safe optimization surface, not an autogenerated guess.

### 3. Probes

Atoms need deterministic verification evidence.

Probe modules under `src/sciona/probes/**` must:

- point to real atom modules and wrapper symbols
- keep probe records structurally valid
- stay cheap and deterministic by default

If a repo uses witness re-export probe modules, those re-exports must target
real witness modules.

### 4. Documentation And Dejargonization

Public atom docs must be readable outside the immediate source discipline.

Required bar:

- no placeholder prose
- no implementation-noise descriptions
- public summaries stay faithful to the real wrapper semantics
- heuristic assets include `dejargonized_summary` or equivalent plain-language
  explanation where the schema supports it

The dejargonization script is a heuristic gate. Review still matters. A passing
score is not permission to write vague or misleading prose.

### 5. Scholarly References And Provenance

When authoritative upstream sources exist, provenance must be reviewable.

`references.json` sidecars should:

- map atom identities to concrete source records
- include at least one non-empty reference record per covered atom
- preserve reviewer notes and confidence when the schema supports them

Broad or hand-wavy provenance is not enough for a merge-ready contribution.

### 6. Heuristics

Heuristic metadata and family heuristic registries must be explicit and
auditable.

Required bar:

- stable heuristic identifiers
- plain-language summaries
- action semantics and uncertainty notes
- provenance/reference notes
- maintainer ownership

Heuristics should explain what evidence means and what downstream action it can
justify. They should not be vague scores without interpretation.

## Expected Repo Layout

The sibling repos do not all share one exact shape, but new work should follow
the existing namespace layout:

```text
src/sciona/atoms/...
src/sciona/probes/...
data/hyperparams/manifest.json
data/heuristics/...
tests/...
```

Per-atom directories remain preferred for complex families because they scale
better for witnesses, references, matches, and uncertainty sidecars.

## Review Checklist

Before merge, reviewers should be able to answer yes to these questions:

1. Is the public interface typed, honest, and documented?
2. If this is a witness-backed wrapper, are contracts and witness bindings
   coherent?
3. Are tunable parameters reviewed rather than guessed?
4. Is there deterministic probe coverage or an explicit probe catalog?
5. Is the documentation understandable outside the narrow source jargon?
6. Are references/provenance explicit where sources exist?
7. Are heuristic claims interpretable, reviewable, and tied to uncertainty?

If the answer is no to any of the above, the PR is not ready.

## Shared Scripts

This repo ships the shared baseline verifier:

- `scripts/verify_contribution_rules.py`
- `scripts/validate_dejargon.py`

They are intentionally structural and deterministic. Domain repos should extend
them with local tests rather than bypass them.
